# Audit log format

## Purpose

The audit log is the tamper-evident record of every agent decision and
tool invocation during an incident run. It is the primary artifact for
post-incident review, the input the `report_emit` gate inspects before
it allows a finalize to happen, and the payload attached to the
`IncidentBundle` export planned for Phase 3.7. A finalized incident that
lacks a corresponding `audit.jsonl` has no chain of custody and is, for
APTWatcher's purposes, unshippable.

This document is the wire-level specification. The higher-level
rationale lives in `../architecture/audit-logging.md`; the present note
is what a reviewer or a consumer tool needs to parse the file
byte-for-byte.

## Wire format

- **Encoding:** UTF-8, no BOM.
- **Framing:** JSON-Lines (`.jsonl`). Exactly one JSON object per line,
  terminated by `\n`. No multi-line objects, no trailing commas, no
  outer array.
- **Line boundaries:** `\n` only; the logger never emits `\r\n`.
- **File path:** `logs/<incident_id>/audit.jsonl`. The directory is
  created on logger construction; one incident, one file, for the full
  run. No rotation mid-incident.
- **Append-only:** the file is opened with `mode="a"` on every write
  and fsync'd before the handle closes. Prior lines are never
  rewritten. Any after-the-fact amendment is represented by a new
  `audit_edit` event, never by mutating a previous line.

The JSONL choice is deliberate: every line is a standalone record that
`jq`, `grep`, and `awk` can consume without schema awareness, and a
torn write at the end of the file loses at most the last event instead
of corrupting the archive.

## Common envelope

Every event — regardless of type — carries this envelope:

```json
{
  "event_type": "tool_call",
  "incident_id": "s01-2026-04-19-1523",
  "correlation_id": "7a1f3c9e2b6d4a08a3d2e4f590c11a77",
  "timestamp": "2026-04-19T15:28:41.512034+00:00",
  "payload": {}
}
```

Field semantics:

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `event_type` | string (one of `AuditEventType`) | yes | See catalog below |
| `incident_id` | string | yes | Stable for the whole run; matches the run directory |
| `correlation_id` | string \| null | no | Pairs start/end `tool_call` events and links findings to the call that produced them |
| `timestamp` | string (ISO-8601, UTC, microsecond precision) | yes | Produced by `core.types.utcnow()`; always emits a `+00:00` offset |
| `payload` | object | yes | Event-type-specific content; may be `{}` but is never omitted |

The `AuditEvent` pydantic model in `src/core/types.py` forbids extra
top-level keys (`extra="forbid"`), so no consumer should rely on
undocumented fields appearing.

## Event type catalog

`AuditEventType` (literal in `src/core/types.py`) currently enumerates
thirteen values. The subsections below cover every one. Examples are
hand-rolled and meant to parse; timestamps and UUIDs are illustrative.

### `run_start`

Written automatically when `AuditLogger` is used as a context manager
(`with AuditLogger(incident_id) as audit:`). It is always the first
line in the file.

```json
{"event_type":"run_start","incident_id":"s01-2026-04-19-1523","correlation_id":null,"timestamp":"2026-04-19T15:23:00.000001+00:00","payload":{"incident_id":"s01-2026-04-19-1523"}}
```

### `run_end`

Written by the same context manager's `__exit__`. Always last. Emitted
even if the run raised — the `payload.error` field carries
`"<ExcType>: <message>"` on crash and `null` on clean exit.

```json
{"event_type":"run_end","incident_id":"s01-2026-04-19-1523","correlation_id":null,"timestamp":"2026-04-19T15:51:12.881004+00:00","payload":{"incident_id":"s01-2026-04-19-1523","error":null}}
```

### `preflight`

Emitted once, early in the run, by `src/agent_extension/cli.py` (see
`cli.py:508`). Payload is the JSON dump of the `PreflightReport`
model: tool inventory, evidence manifest, tier config, warnings, and
the overall `ok` bit.

```json
{"event_type":"preflight","incident_id":"s01-2026-04-19-1523","correlation_id":null,"timestamp":"2026-04-19T15:23:02.104521+00:00","payload":{"profile":"windows-host-triage","ok":true,"missing_required":[],"warnings":[]}}
```

### `tool_call`

Every SIFT tool invocation writes two events that share a
`correlation_id` generated by `core.sift.runner.run_tool` as a
`uuid.uuid4().hex`. The `phase` discriminator is the payload key every
consumer must branch on.

**Phase `start`** — written before the subprocess is spawned:

```json
{"event_type":"tool_call","incident_id":"s01-2026-04-19-1523","correlation_id":"7a1f3c9e2b6d4a08a3d2e4f590c11a77","timestamp":"2026-04-19T15:28:41.100000+00:00","payload":{"phase":"start","tool":"volatility3","argv":["/usr/bin/vol3","-f","mem.raw","windows.malfind"],"cwd":null}}
```

**Phase `end`** — written after the subprocess returns or times out,
with the same `correlation_id`:

```json
{"event_type":"tool_call","incident_id":"s01-2026-04-19-1523","correlation_id":"7a1f3c9e2b6d4a08a3d2e4f590c11a77","timestamp":"2026-04-19T15:28:45.821034+00:00","payload":{"phase":"end","tool":"volatility3","returncode":0,"duration_seconds":4.72,"timed_out":false,"stdout_bytes":18204,"stderr_bytes":0}}
```

The planner phase in `src/core/agent_loop.py:_plan` also writes a
`tool_call` event with `payload.phase == "plan"` and
`payload.tool == "planner"`. Consumers that filter on `phase` should
be aware of this third phase value.

### `self_correction`

Emitted by `AgentLoop._verify_and_correct` (see `agent_loop.py:308`)
after every verify+correct cycle, including the trailing one a
zero-finding run performs. The payload carries the verifier's issues,
the self-corrector's decisions, and the `replan` flag.

```json
{"event_type":"self_correction","incident_id":"s01-2026-04-19-1523","correlation_id":null,"timestamp":"2026-04-19T15:48:02.443101+00:00","payload":{"iteration":3,"issues":[{"severity":"block","rule":"rule1_evidence_required","finding_id":"F-014","detail":"Finding has zero citations; report emitter will refuse."}],"resolved":[],"dropped":["F-014"],"replan":false,"notes":"null-self-corrector: drop blocking findings, do not replan."}}
```

### `report_emit`

Emitted by `AgentLoop.finalize()` (see `agent_loop.py:260`). The method
refuses to write this event unless
`state.self_correction_done_for_current_findings` is true, which is the
architectural form of the self-correction gate.

```json
{"event_type":"report_emit","incident_id":"s01-2026-04-19-1523","correlation_id":null,"timestamp":"2026-04-19T15:48:03.009822+00:00","payload":{"incident_id":"s01-2026-04-19-1523","finding_count":7,"iterations":3}}
```

### `llm_call`

Emitted by strategy classes that drive the LLM — today,
`LLMPlanner._emit_audit` in `src/core/strategies/llm_planner.py:277`.
The payload is intentionally compact: no prompt text, no completion
text, only the model's structural outputs (reasoning summary, the step
IDs it asked for, the finalize flag, and any parse error that forced
the planner into defensive finalize mode).

```json
{"event_type":"llm_call","incident_id":"s01-2026-04-19-1523","correlation_id":null,"timestamp":"2026-04-19T15:24:11.320005+00:00","payload":{"tool":"llm_planner","iteration":0,"finalize":false,"step_count":3,"step_ids":["s1","s2","s3"],"reasoning":"Scope host triage; start with preflight+registry+EVTX."}}
```

Token counts, the `stop_reason`, the `model_id`, and a
`prompt_digest` are all plausible additions — the `ModelCompletion`
type in `core/llm.py` carries `stop_reason` and a usage dict already
— but none of them are written to the audit log today. See "Code gaps"
below.

### `sift_update_consent`

Emitted by `src/core/sift/update.py:run_sift_update` (see line 122)
immediately BEFORE the `tool_call` start event that wraps the
`apt-get` invocation. The raw consent token is never written: only
its length and the presence flag. The allow-listed package set and
`dry_run` flag are captured so a reviewer can confirm what the
operator consented to.

```json
{"event_type":"sift_update_consent","incident_id":"s01-2026-04-19-1523","correlation_id":null,"timestamp":"2026-04-19T15:25:00.772010+00:00","payload":{"consent_token_present":true,"consent_token_length":42,"packages":["python3-plaso","python3-volatility3","yara"],"dry_run":true}}
```

### Other declared types

The `AuditEventType` literal also declares `finding`,
`claim_verification`, `containment_action`, `ticket_operation`, and
`audit_edit`. Each has a documented intent in
`../architecture/audit-logging.md`, but they are not yet emitted by
any code path in `src/core/`. They are reserved for Phase 3 (Tier 2
ticketing), Phase 3.3 (containment), and Phase 3.7 (incident-bundle
amendments). New code paths that emit them MUST follow the common
envelope above.

## Event ordering guarantees

1. **Book-ends.** `run_start` is the first line; `run_end` is the last.
   Both are produced by the `AuditLogger` context manager (`__enter__`
   / `__exit__`) so they are written even if the run raises.
2. **Consent precedes tool call.** For any SIFT update, the
   `sift_update_consent` event MUST appear before the paired
   `tool_call` start. `core.sift.update.run_sift_update` enforces this
   by calling `audit.append("sift_update_consent", ...)` before
   handing control to `run_tool`.
3. **Self-correction precedes report emit.** `AgentLoop.finalize()`
   raises `ReportEmitError` unless
   `self_correction_done_for_current_findings` is true — meaning a
   `self_correction` event has been written since the finding set last
   changed. Any `add_finding()` invalidates the flag (see
   `agent_loop.py:271`), forcing another correction pass.
4. **Paired tool calls share a correlation id.** `run_tool` generates
   a single `uuid.uuid4().hex` and uses it for both the `start` and
   `end` audit appends. Consumers reconstructing timing join on that
   id. The pairing invariant is: for every start there is exactly one
   end; a missing end implies a crash between the two (the `run_end`
   event's `error` field will then be non-null).
5. **Planner events intermix with tool calls.** The
   `tool_call phase=plan` event is emitted inside
   `AgentLoop._plan` on every iteration. Between any two
   `phase=plan` events, zero or more `phase=start`/`phase=end` pairs
   may appear (one pair per executed step).

## Redaction rules

`AuditLogger._redact` (see `src/core/audit.py:43`) walks every payload
dict recursively. Any key whose lower-cased name is in the redaction
set below has its value replaced with the literal string
`"<redacted>"` before the line is serialized. List values are walked
too.

Sensitive-key set (from `_REDACT_KEYS` in `audit.py`):

- `api_key`
- `apikey`
- `token`
- `bearer`
- `password`
- `secret`
- `client_secret`
- `authorization`

Additional rules enforced by the call sites (not by `_redact`):

- **Explicit consent tokens.** `sift_update_consent` payloads log
  `consent_token_present: true` and `consent_token_length: N`, never
  the raw value. The update wrapper strips the token from its
  internal state before building the payload.
- **Tier 1 intel adapter keys.** HTTP provider adapters pull bearer
  credentials from environment variables; the adapter never includes
  the credential in the tool invocation payload. Audit payloads
  reference the env-var name (e.g. `env=APTWATCH_API_KEY`) only.
- **LLM prompts and completions.** The current `llm_call` payload
  summarizes structure (step count, finalize flag, trimmed reasoning
  capped at 2000 chars) and does not include raw prompts or
  completions. A future `prompt_digest` field (SHA-256 of the
  canonicalized prompt) would preserve verifiability without
  leaking content; see "Code gaps".

Redaction happens inside the logger before `model_dump_json`, so a
malformed caller cannot accidentally bypass it. It does NOT attempt to
redact unstructured content (free-text payload values that happen to
contain a token) — the rule is "don't hand secrets to the payload in
the first place, and if you do, put them under a redacted-keyed
name".

## Durability

`AuditLogger._write_line` (see `audit.py:157`) implements fsync-per-line
semantics:

1. Acquire the process-local `threading.Lock` so concurrent tool
   calls cannot interleave half-lines.
2. Open the file in append mode, write the serialized event plus a
   trailing `\n`, call `f.flush()`, call `os.fsync(f.fileno())`,
   close.
3. Swallow `OSError` on `fsync` (some pseudo-filesystems do not
   support it) rather than crashing the run.

The reason is crash safety: a SIGKILL between two events must leave a
file whose prefix is valid JSONL. Any event that appears on disk has
already been fsync'd, which means the reviewer can treat the audit
log as an honest prefix even after a hard crash. The cost — one fsync
per event — is acceptable because an incident produces on the order of
a few thousand events, not millions.

## Consumers

- **`aptwatcher run`** (`src/agent_extension/cli.py`) owns the
  `AuditLogger` lifetime. It opens the log, runs the agent loop inside
  the context manager, and closes it on exit.
- **`AgentLoop.finalize()`** uses
  `self_correction_done_for_current_findings` (set by the
  verify+correct step) as the gate. Conceptually it is reading
  "was there a recent `self_correction` event for the current findings
  set?" — today the check is an in-memory flag; a future hardening is
  to replay `audit.find("self_correction")` and compare to the
  current finding-id set (see "Code gaps").
- **`IncidentBundle` export (Phase 3.7)** will attach the entire
  `audit.jsonl` and its SHA-256 to the signed bundle. Recipients can
  recompute the digest and verify the sender's Ed25519 signature
  covers both findings and audit trail.
- **Operator review.** The format is intentionally `jq`-friendly:
  `jq 'select(.event_type=="tool_call" and .payload.phase=="end")'`
  is the day-one triage command.

## Non-goals

- **Not a debug log.** Stack traces, timing probes, and `print`
  output do not belong here. Use stderr or a separate structured
  logger.
- **Not a metrics stream.** The file is not sampled, not aggregated,
  and not streamed to Prometheus. Metrics tooling reads operational
  logs, not the audit log.
- **Not human-readable raw.** Expect to run `jq` or a rendering tool.
  A cat-friendly summary layer is a separate concern (Phase 4).
- **Not mutable.** Never edit a prior line. Amendments are new
  `audit_edit` events.
- **Not a privacy layer.** Redaction covers credential-shaped keys
  only; it is not a GDPR mechanism. Evidence-level PII handling is
  the responsibility of the SIFT wrapper that produced the evidence.

## Versioning

The envelope in the current implementation does NOT carry a
`schema_version` field. The `AuditEvent` model has five fixed keys
(`event_type`, `incident_id`, `correlation_id`, `timestamp`,
`payload`) and no version indicator. This is a gap: future format
evolutions (adding `model_id` to `llm_call`, new event types,
structural changes to the envelope) cannot be negotiated by
consumers without a version to branch on.

**Recommendation:** add `"schema_version": "1.0"` to the envelope as
the first-class sixth field, emitted on every line. Bump to `"1.1"`
when backward-compatible fields are added (new optional payload
keys); bump to `"2.0"` on breaking shape changes. Consumers MUST
tolerate unknown `schema_version` values by falling back to
best-effort parsing of known keys.

## Code gaps

The following deltas between the spec and the implementation are known
and tracked:

1. **No `schema_version` on the envelope.** See §Versioning.
2. **`llm_call` payload is thin.** `stop_reason` and token usage are
   available on `ModelCompletion` (`src/core/llm.py:103`) but are not
   copied into the audit payload. `model_id` and a `prompt_digest` are
   not computed today. Adding them would make LLM cost reconciliation
   and prompt-replay verification tractable.
3. **`finding`, `claim_verification`, `containment_action`,
   `ticket_operation`, `audit_edit`** are declared in the literal but
   not emitted anywhere in `src/core/`. They are reserved for later
   phases; grep consumers should not assume they exist in MVP logs.
4. **`self_correction` gate is in-memory.** `AgentLoop.finalize()`
   trusts a boolean on the state object rather than rereading the
   log. A process that bypassed the loop and appended findings
   through a side channel could, in theory, emit a report without a
   correction event. Hardening: replay `audit.find("self_correction")`
   and compare its implied finding set to the current one.
5. **`preflight` payload shape is `PreflightReport.model_dump`.**
   It is effectively stable (the pydantic model is pinned) but the
   wire spec should pin it explicitly: enumerate every field the
   consumer can expect. The `../architecture/audit-logging.md` page
   shows an older shape with `tool_inventory` as a flat dict; the
   current model emits it as a list of `ToolVersion` objects.

None of these gaps block MVP. All five deserve follow-up tickets.
